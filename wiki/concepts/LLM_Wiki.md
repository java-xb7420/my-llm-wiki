# LLM Wiki

## 概述

LLM Wiki 是前 OpenAI 科学家 Andrej Karpathy 于 2026 年 4 月提出的一种利用大语言模型构建个人知识库的范式。核心思路是让 LLM **增量构建并维护一份持久化的 Wiki**，而非传统的 RAG（检索增强生成）方式。

## 核心理念：Wiki vs RAG

传统 RAG 的问题：用户上传一批文件，LLM 在查询时检索相关片段并生成答案。每次回答都要从零重新发现知识，没有积累。需要综合五份文档的复杂问题，模型每次都要重新查找、拼接碎片。

LLM Wiki 的不同之处：**LLM 在原始资料与用户之间，逐步构建并维护一套结构化、相互关联的 Markdown 文件集合**。添加新资料时，LLM 不只是索引，而是阅读内容、提取关键信息，并整合进现有 Wiki——更新实体页面、修订主题摘要、标注新旧数据矛盾。知识编译一次并持续更新，而非每次查询重新推导。

关键区别：**Wiki 是持久化、可不断沉淀的成果**。交叉引用已预先建立，矛盾点已标注，整合已涵盖所有已读资料。每新增一份资料、每提出一个问题，Wiki 都更加丰富。

## 三层架构

```
┌─────────────────┐
│  Schema 层      │  规范文档（CLAUDE.md / AGENTS.md）
│                 │  定义结构、约定、工作流程
├─────────────────┤
│  Wiki 层        │  LLM 生成的 Markdown 文件
│                 │  摘要、实体、概念、对比、整合
├─────────────────┤
│  原始资料层     │  文章、论文、图片、数据文件
│                 │  不可修改，LLM 只读
└─────────────────┘
```

- **原始资料层**：用户整理的源文档集合，不可变，是事实依据。
- **Wiki 层**：LLM 生成的 Markdown 文件目录，LLM 完全拥有该层——创建、更新、维护交叉引用。
- **规范层**：告诉 LLM 如何维护 Wiki 的配置文档，是 LLM 成为专业 Wiki 维护者而非普通聊天机器人的关键。

## 三大操作

### 1. 资料录入（Ingest）

将新资料放入原始资料文件夹，LLM 进行处理：
- 阅读资料，提取关键信息
- 在 Wiki 中创建摘要页面
- 更新索引和相关实体/概念页面
- 追加日志记录

单份资料可能涉及 10-15 个 Wiki 页面。可逐份录入保持参与，也可批量处理。

### 2. 知识查询（Query）

向 Wiki 提问，LLM 搜索相关页面、阅读内容并生成带引用的答案。答案形式可多样：Markdown 页面、对比表、幻灯片（Marp）、图表等。

核心要点：**优质答案可作为新页面存回 Wiki**。对比分析、研究结论、关联发现不应仅存于聊天记录，而应在 Wiki 中沉淀。

### 3. 质量校验（Lint）

定期对 Wiki 进行健康检查：
- 页面间的矛盾内容
- 被新资料推翻的过时结论
- 无入站链接的孤立页面
- 提及但未建页的重要概念
- 缺失的交叉引用
- 可通过网络搜索补充的数据缺口

## 索引与日志

| 文件 | 功能 |
|------|------|
| `index.md` | 内容总目录，按分类列出所有页面链接和摘要，LLM 查询时先读索引再深入 |
| `log.md` | 时间顺序的操作日志，仅追加。固定前缀格式使其可用 Unix 工具解析 |

在中等规模场景下（约 100 份资料、数百个页面），索引文件即可满足检索需求，无需搭建向量嵌入基础设施。

## 应用场景

| 场景 | 描述 |
|------|------|
| 个人用途 | 目标、健康、心理、自我成长，逐步构建结构化个人档案 |
| 学术研究 | 数周/数月深耕主题，搭建内容全面的专题 Wiki |
| 书籍阅读 | 逐章整理，为角色、主题、情节建页，形成配套知识库 |
| 企业/团队 | LLM 维护的内部 Wiki，数据源包括对话、会议记录、项目文档 |
| 其他 | 竞品分析、尽职调查、旅行规划、课程笔记等 |

## 实用技巧

- **Obsidian Web Clipper**：浏览器插件，将网页文章转为 Markdown
- **本地保存图片**：Obsidian 中设置附件路径并绑定下载快捷键
- **图谱视图**：Obsidian 的 Graph View 查看 Wiki 结构和关联
- **Marp**：Markdown 幻灯片格式，Obsidian 有对应插件
- **Dataview**：Obsidian 插件，查询页面前置元数据生成动态表格
- **qmd**：本地 Markdown 搜索引擎，支持 BM25/向量混合检索和 LLM 重排序

## 范式优势

维护知识库的繁琐之处不是阅读与思考，而是**日常管理**：更新交叉引用、同步摘要、标注矛盾、保证一致性。人类因维护成本超过价值而放弃 Wiki，LLM 则不会厌倦、不会遗漏、一次可修改十几个文件。维护成本近乎为零，Wiki 得以持续完善。

- **人类的工作**：筛选资料、主导分析、提出问题、思考内涵
- **LLM 的工作**：总结、交叉引用、归档、维护

该理念与范内瓦·布什 1945 年提出的 Memex（麦克斯存储器）精神相通：个人化知识存储，文档间存在关联路径。布什未能解决的维护问题，如今由 LLM 实现。
