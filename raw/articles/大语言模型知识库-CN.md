# 大语言模型知识库

一种利用大语言模型搭建个人知识库的范式。

本文档为思路说明，可直接复制粘贴到你使用的大语言模型智能体中（如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。其目的是传递核心思路，具体细节将由智能体与你协作完善。

---

## 核心理念

大多数人使用大语言模型处理文档的方式都是检索增强生成（RAG）：上传一批文件，大语言模型在查询时检索相关片段并生成答案。这种方式可行，但模型每次回答都要从零重新挖掘知识，无法实现积累。若提出需要综合五份文档的复杂问题，模型每次都要重新查找、拼接相关内容，没有任何沉淀。NotebookLM、ChatGPT 文件上传功能以及多数检索增强生成系统均采用此模式。

本文提出的思路截然不同。大语言模型不再仅在查询时从原始文档中检索内容，而是**逐步构建并维护一份持久化的知识库**—— 一套位于你与原始资料之间、结构化且相互关联的 Markdown 文件集合。添加新资料时，模型不只是为后续检索建立索引，而是阅读内容、提取关键信息，并将其整合进现有知识库：更新实体页面、修订主题摘要、标注新数据与旧结论的矛盾之处，强化或修正不断完善的知识体系。知识只需编译一次并持续更新，而非每次查询都重新推导。

核心区别在于：**这份知识库是持久化、可不断沉淀的成果**。交叉引用已预先建立，矛盾点已标注，知识整合已涵盖你阅读过的所有内容。每新增一份资料、每提出一个问题，知识库都会变得更加丰富。

你几乎无需亲自编写内容，全部由大语言模型撰写与维护。你只需负责搜集资料、探索研究、提出精准问题；模型则承担所有繁琐工作：总结、交叉引用、分类归档、日常维护，让知识库长期保持实用价值。实际使用中，我会一边打开大语言模型智能体，一边打开 Obsidian。模型根据对话内容编辑文档，我实时浏览结果 —— 跳转链接、查看图谱、阅读更新页面。此时 Obsidian 是集成开发环境，大语言模型是开发者，知识库则是代码库。

该范式适用于多种场景，举例如下：

- **个人用途**：记录目标、健康、心理、自我成长内容，整理日记、文章、播客笔记，逐步构建结构化的个人档案。

- **学术研究**：数周乃至数月深耕某一主题，阅读论文、文章、报告，逐步搭建内容全面、观点不断完善的专题知识库。

- **书籍阅读**：逐章整理内容，为角色、主题、情节线索及其关联关系建立页面，读完后形成一份详实的配套知识库。可类比托尔金网关等粉丝维基 —— 由社区志愿者耗时多年搭建，包含数千个关联角色、地点、事件、语言的页面。借助大语言模型，你在阅读时也能个人搭建同类知识库，由模型完成所有交叉引用与维护工作。

- **企业 / 团队用途**：由大语言模型维护的内部知识库，数据源包括 Slack 对话、会议记录、项目文档、客户通话内容，可由人工审核更新内容。模型承担团队无人愿意做的维护工作，确保知识库实时更新。

- **其他场景**：竞品分析、尽职调查、旅行规划、课程笔记、爱好深度研究等，所有需要长期积累知识并希望内容系统化而非零散存放的场景。

## 架构设计

整体分为三层：

1. **原始资料层**
你整理的各类源文档，包括文章、论文、图片、数据文件。该层内容不可修改，大语言模型仅读取不编辑，是你的事实依据来源。

2. **知识库层**
由大语言模型生成的 Markdown 文件目录，包含摘要、实体页面、概念页面、对比内容、概述、知识整合等。该层完全由模型负责：创建页面、根据新资料更新内容、维护交叉引用、保证内容一致性。你负责阅读，模型负责编写。

3. **规范层**
一份配置文档（如适配 Claude Code 的 `CLAUDE\.md`、适配 Codex 的 `AGENTS\.md`），告知模型知识库的结构、编写规范，以及录入资料、回答问题、维护知识库的工作流程。这是核心配置文件，能让模型成为专业的知识库维护者，而非普通聊天机器人。你可与模型共同优化这份规范，适配你的使用领域。

## 操作流程

### 1\. 资料录入

将新资料添加至原始资料文件夹，指令模型进行处理。示例流程：
模型阅读资料→与你沟通核心要点→在知识库中创建摘要页面→更新索引→同步更新知识库中相关实体与概念页面→在日志中添加记录。
单份资料可能会关联 10–15 个知识库页面。我个人习惯逐份录入并全程参与：阅读摘要、检查更新、引导模型突出重点内容；你也可批量录入多份资料，减少人工干预。可根据自身习惯制定工作流程，并记录在规范文档中。

### 2\. 知识查询

向知识库提出问题，模型检索相关页面、阅读内容并生成带引用的答案。答案形式可按需调整：Markdown 页面、对比表格、幻灯片（Marp）、图表（matplotlib）、画布等。
核心要点：优质答案可作为新页面存入知识库。你提出的对比分析、研究结论、关联发现等内容，不应仅留存于聊天记录。通过这种方式，你的探索成果会和录入的资料一样，在知识库中持续沉淀。

### 3\. 质量校验

定期指令模型对知识库进行健康检查，排查内容包括：
页面间的矛盾内容、被新资料推翻的过时结论、无入站链接的孤立页面、提及但未独立建页的重要概念、缺失的交叉引用、可通过网络搜索补充的数据缺口。
模型可主动推荐待研究的新问题与待查找的新资料，保障知识库在扩容过程中保持规范可用。

## 索引与日志

两份特殊文件辅助模型与你管理不断扩容的知识库，功能各有侧重：

- **`index\.md`****（内容索引）**
知识库内容总目录，列出所有页面链接、单行摘要，可附加日期、资料数量等元数据，按实体、概念、资料等分类整理。模型每次录入资料都会更新该索引。查询时，模型先读取索引定位相关页面，再深入查阅内容。在中等规模场景下（约 100 份资料、数百个页面）效果极佳，无需搭建基于向量嵌入的检索增强生成基础设施。

- **`log\.md`****（操作日志）**
按时间顺序记录所有操作，仅追加不修改，包括资料录入、查询、质量校验。实用技巧：每条记录以固定前缀开头（如 `\#\# \[2026\-04\-02\] ingest \| 文章标题`），可通过 Unix 工具直接解析，例如执行 `grep \&\#34;^\#\# \[\&\#34; log\.md \| tail \-5` 即可查看最近 5 条操作。日志记录知识库的演进时间线，帮助模型了解近期操作内容。

## 可选：命令行工具

后续可搭建小型工具，提升模型操作知识库的效率。最常用的是知识库搜索引擎：小规模场景下索引文件即可满足需求，知识库扩容后则需要专业检索工具。`qmd` 是优质选择 —— 面向本地 Markdown 文件的搜索引擎，支持 BM25 / 向量混合检索与大语言模型重排序，全程本地运行；同时提供命令行接口（模型可直接调用）与 MCP 服务（模型可作为原生工具使用）。你也可自行搭建简易工具，有需求时可让模型协助编写基础检索脚本。

## 使用技巧

- Obsidian 网页剪藏插件可将网页文章转为 Markdown 格式，快速将资料存入原始文件夹。

- 本地保存图片：在 Obsidian 设置→文件与链接中，将「附件文件夹路径」设为固定目录（如 `raw/assets/`）；在设置→快捷键中搜索「下载」，为「下载当前文件附件」绑定快捷键（如 Ctrl\+Shift\+D）。剪藏文章后按下快捷键，所有图片将保存至本地。该操作可选但实用，模型可直接查看引用图片，避免链接失效。注：大语言模型无法一次性读取含内嵌图片的 Markdown，可先让模型读取文本，再单独查看部分或全部图片补充信息，流程稍繁琐但效果良好。

- Obsidian 图谱视图是查看知识库结构的最佳方式，可清晰看到页面关联、核心节点与孤立页面。

- Marp 是基于 Markdown 的幻灯片格式，Obsidian 有对应插件，可直接从知识库内容生成演示文稿。

- Dataview 是 Obsidian 插件，可查询页面前置元数据。若模型为知识库页面添加 YAML 前置信息（标签、日期、资料数量等），Dataview 可生成动态表格与列表。

- 知识库本质是存放 Markdown 文件的 Git 仓库，天然支持版本历史、分支管理与协作功能。

## 范式优势

维护知识库的繁琐之处并非阅读与思考，而是**日常管理工作**：更新交叉引用、同步摘要内容、标注新旧数据矛盾、保证数十个页面的一致性。人类因维护成本远超价值而放弃维基，大语言模型则不会厌倦、不会遗漏引用更新，可一次性修改多个文件。维护成本近乎为零，知识库得以持续完善。

人类的工作是筛选资料、主导分析、提出优质问题、思考知识内涵；其余工作均由大语言模型完成。

该理念与范内瓦・布什 1945 年提出的「麦克斯存储器」精神相通：个人化、可整理的知识存储库，文档间存在关联路径。布什的构想更贴近本范式，而非如今的互联网 —— 私密、主动整理，文档间的关联与文档本身同等重要。他当年未能解决的维护问题，如今由大语言模型实现。

## 备注

本文档侧重理念阐述，不涉及具体实现方案。具体目录结构、规范约定、页面格式、工具选择等，均取决于你的使用领域、个人偏好与所选大语言模型。文中所有内容均为可选模块化设计，按需取用即可。例如：仅使用文本资料则无需处理图片；知识库规模较小时仅靠索引文件即可，无需搜索引擎；无需幻灯片可仅保留 Markdown 页面；也可自定义其他输出格式。
正确使用方式是将本文档分享给你的大语言模型智能体，协作搭建适配自身需求的版本。本文档仅用于传递该范式思路，剩余细节可由模型自主实现。
